# Дорожная карта проекта

Этот документ — живой план развития проекта. Заполняем по мере уточнения требований и согласований.

## 1) Обзор
- Название проекта: Media Generator
- Ценность для пользователей: Быстрый запуск тематических Telegram-каналов со сводками новостей на базе публикаций из выбранных каналов; минимизация ручной работы за счёт автоматической саммаризации и модерации.
- Целевые аудитории: Я и моя команда (возможность множить инстансы под разные темы).
- Бизнес-цели (3–5):
  1) Сократить время подготовки контента для сводок в 5–10 раз.
  2) Упростить запуск новых тематических каналов (развёртывание «по шаблону»).
  3) Обеспечить стабильную публикацию периодических сводок с контролем качества.
  4) Снизить стоимость на модерацию за счёт удобных инструментов и ролей.

## 2) Этапы и вехи
- Этап 0: Подготовка (аналитика, стек, архитектура, инфраструктура)
- Этап 1: MVP (минимально жизнеспособный продукт) — срок 2 недели
- Этап 2: Beta (обратная связь, улучшения, стабильность)
- Этап 3: Production (масштабирование, мониторинг, поддержка)

## 4) Этап 1 — MVP
- Ключевые фичи MVP:
  1. Сервис «Collector»: подключение Telegram-аккаунта (MTProto) к выбранным каналам, сбор новых постов, авто-саммаризация (LLM), постановка в очередь модерации.
  2. Модерация в Telegram-боте: роли Admin/Editor по tg-id из конфига; Editor утверждает/отклоняет краткие саммари.
  3. Сервис «Aggregator»: формирование периодической сводки (например, дневной) на основе утверждённых саммари и публикация в целевой канал.
  4. Развёртывание через Docker Compose, конфигурация через ENV.
- Пользовательские сценарии (User Stories):
  - Как Admin, я хочу подключать по MTProto Telegram-аккаунт, чтобы собирать посты из каналов.
  - Как Admin, я хочу добавлять/удалять каналы для мониторинга, чтобы управлять источниками.
  - Как Editor, я хочу быстро утверждать или отклонять саммари, чтобы поддерживать качество контента.
  - Как Admin, я хочу получать готовую сводку по расписанию в наш канал, чтобы не тратить время на ручную компоновку.
  - Как User, я хочу видеть сводки в канале, чтобы быть в курсе новостей.
- Критерии приемки (DoD):
  - Collector обрабатывает новые посты из ≤100 каналов и создаёт саммари.
  - Модерация работает через бота с ограничением доступа по tg-id.
  - Aggregator формирует и публикует сводку по расписанию.
  - Тестовый прогон и базовая документация готовы; нет блокирующих багов (P0/P1).
- Сроки и вехи (2 недели):
  - М1 (день 3–4): черновой Collector, базовая саммаризация, бот-модерация без всех нюансов.
  - М2 (день 7–8): Aggregator с простым шаблоном сводки, публикация в канал.
  - М3 (день 14): Полный MVP: Docker Compose, конфигурация, роли, отладка.

## 5) Нефункциональные требования
- Доступ/Безопасность:
  - Настройки и админ-команды доступны только tg-id из конфига.
  - Роли Admin/Editor: Editor видит и утверждает саммари; Admin управляет конфигурацией.
- Производительность: одновременный мониторинг до 100 каналов на один инстанс; p95 обработки нового поста ≤ 60 c (включая LLM-вызов).
- Надёжность/Доступность: устойчивость к временным ошибкам Telegram/LLM (ретраи, очереди), логирование и алерты на фатальные сбои.
- Масштабирование: горизонтальное — отдельные инстансы под темы; возможность вынести БД в общий кластер на следующих этапах.
- Соответствие: безопасное хранение секретов (ENV/secret manager), без коммита ключей.

## 6) Архитектура (высокоуровнево)
- Сервисы: два контейнера — Collector (MTProto + модерация-бот) и Aggregator (составление сводок + публикация).
- Язык/Фреймворки: Python 3.11; Telethon (MTProto), Aiogram (бот), FastAPI (сервисные эндпоинты/health), APScheduler (расписание), SQLAlchemy/SQLModel (ORM).
- Хранилище: PostgreSQL (общая БД для двух сервисов в MVP); Redis (опционально позже для очередей).
- Интеграции: Telegram MTProto (учётка), Telegram Bot API (бот), LLM-провайдер по OpenAI-совместимому API.

## 7) Риски и предположения
- Риски:
  - Ограничения Telegram API — следить за rate limit при чтении каналов для стабильной работы.
  - Качество и стабильность Gemini CLI — кэширование результатов, мониторинг доступности API, контроль квот бесплатного доступа.
  - Качество саммари — итерация промптов, ручная модерация.
- Предположения:
  - До 100 каналов достаточно для одного инстанса; периодичность сводки — дневная по умолчанию.

## 8) Метрики успеха
- Среднее время от появления поста до готового саммари и утверждения.
- Доля отклонённых саммари (индикатор качества промптов/источников).
- Стабильность публикаций сводок по расписанию, p95 времени формирования.

## 9) План релизов
- v0.1.0 — «Prototype»: Collector читает каналы и создаёт черновые саммари; бот принимает approve/reject; ручной пост в тестовый канал.
- v0.2.0 — «MVP»: Aggregator формирует дневную сводку и публикует автоматически; Docker Compose; роли и доступ по tg-id; базовая наблюдаемость.
- v0.3.0 — «Beta»: Улучшение качества саммари, настройка шаблонов сводок, ретраи/очереди, простые графики метрик.
- v1.0.0 — «Production»: Тюнинг масштабирования, алерты, финальная документация и отладка.
